# AI-Powered API Documentation & Testing Portal

A working portal where users upload route definitions (OpenAPI/Swagger,
FastAPI/Flask source, or freeform text) and instantly get AI-generated
documentation, sample requests/responses, runnable test cases, and
plain-English error explanations for failed calls.

## Architecture

```
User → Streamlit UI → FastAPI backend → Parser → normalized endpoint list
                                       → AI Engine (OpenAI / NVIDIA NIM) → docs, tests, error explanations
                                       → httpx → live "Try it" calls to real APIs
```

| Layer        | Tech                          | File                        |
|--------------|--------------------------------|-----------------------------|
| UI           | Streamlit                     | `frontend/app.py`           |
| Backend/API  | FastAPI                       | `backend/main.py`           |
| Parser       | Python (regex + JSON/YAML)    | `backend/parser.py`         |
| AI           | OpenAI-compatible / NVIDIA NIM| `backend/ai_engine.py`      |
| Starter data | JSON, 100 endpoints           | `data/endpoints.json`       |

## Why AI is used (and where)

The brief asks students to justify the AI/agent logic, not just bolt it on. Three places:

1. **Documentation generation** — turning a sparse schema (`{method, path, request_body}`)
   into a real explanation of *why* a field exists and how to use the endpoint is exactly
   what hand-written docs teams skip under deadline pressure. An LLM reads the shape of
   the schema and writes the missing prose.
2. **Test-case generation** — enumerating edge cases (missing auth, invalid payload,
   not-found id) by hand is repetitive. The AI reasons over the schema to infer which
   edge cases are implied (e.g. a required `email` field → an invalid-email test case)
   and emits runnable pytest snippets.
3. **Error explanation** — this is the "actionable output" requirement. A raw
   `422 Unprocessable Entity` doesn't tell a developer what to *do*. The AI reads the
   endpoint definition + the actual failed response and returns: likely cause, plain
   English explanation, and a concrete suggested fix.

**Important design choice:** every AI feature has a deterministic template-based
fallback (`backend/ai_engine.py`, `_fallback_*` functions) so the whole portal runs
end-to-end with zero API key / no internet — useful for demos/judging where network
access is unreliable. Set `AI_API_KEY` to switch on live LLM calls.

## Setup

```bash
cd api-doc-portal
pip install -r requirements.txt

# Optional: enable live AI (otherwise template fallback is used automatically)
export AI_API_KEY="sk-..."
export AI_BASE_URL="https://api.openai.com/v1"   # or https://integrate.api.nvidia.com/v1 for NIM
export AI_MODEL="gpt-4o-mini"                      # or e.g. meta/llama-3.1-70b-instruct for NIM

# Terminal 1 — backend
uvicorn backend.main:app --reload --port 8000

# Terminal 2 — frontend
streamlit run frontend/app.py
```

Then open the Streamlit URL it prints (usually http://localhost:8501).

## Using the portal

1. **Load data** — three ways:
   - Upload an OpenAPI JSON/YAML file, our own `endpoints.json` schema, or a
     FastAPI/Flask `.py` source file (try `sample_uploads/sample_fastapi_routes.py`).
   - Paste raw JSON or freeform lines like `GET /api/v1/users - list users`.
   - Click "Load starter dataset" to instantly load the 100 bundled sample endpoints.
2. **Pick an endpoint** from the dropdown.
3. **Generate documentation** — plain-English summary, parameter explanations,
   sample request/response, usage notes.
4. **Generate test cases** — happy path, auth failure, validation failure,
   not-found, each with a ready-to-paste pytest snippet.
5. **Try it live** — enter a real base URL and actually call the endpoint;
   failures are auto-explained by the AI.
6. **Explain an error** — paste any status code + response body (from Postman,
   logs, etc.) and get cause + fix, independent of the rest of the flow.

## Dataset

`data/endpoints.json` contains 100 realistic CRUD endpoints (GET list, POST,
GET by id, PUT, DELETE) across 20 resources (users, orders, payments, etc.),
plus a shared error-code reference table. Regenerate it with:

```bash
cd data && python generate_dataset.py
```

## What "actionable output" means here

This portal doesn't stop at "code ran successfully." Every screen ends in a
decision a developer can act on immediately:
- Documentation tells them **how to call the endpoint correctly**.
- Test cases give them **copy-pasteable tests to add to CI**.
- Error explanations tell them **exactly what to change** in their request.

## Notes / limitations

- Data is stored in-memory per session (keyed by a UUID generated by the
  Streamlit app) — restarting the backend clears loaded endpoints. This is
  intentional for a lightweight project scope; swap in a real DB for production.
- The Python-source parser is regex-based and heuristic (detects FastAPI
  `@app.get/post/...` and Flask `@app.route` decorators) — it will not handle
  every possible code style, but covers the common cases.
